# Реактивный гейт command_reroll (решение после броска, pay-on-use) — дизайн

Дата: 2026-06-25
Статус: согласован заказчиком (подход A)
Исполнение: дирижёр GPT 5.5 (orchestrator-40k) → исполнитель GLM 5.2 (executor). GPT обязан
**разобрать этот spec в детальный план** (bite-sized TDD под-задачи) перед делегированием.

## Цель

Сделать command_reroll **реактивным**: решение «перебрасывать ли» принимается **в момент
фактического броска** (как overwatch/go_to_ground), а CP списывается **только если реролл реально
применён**. Это структурно убирает остаточный waste (CP, потраченный на реролл, который не
сработал) — для ВСЕХ алгоритмов (DQN/PPO/AZ/GMZ/SMZ), на уровне движка.

Контекст: движковый гейт «нет цели → нет реролла» (коммит `1e1f4f41`) убрал только no-target
случай. Остаток waste (~60-70%) — «эффект армирован, но нужного броска не случилось / реролл
бесполезен». Его ловит только реактивная модель.

## Текущее поведение (факты из кода)

- Реролл — **пре-коммит**: выбирается в стратагем-окне (головы `strat_command`/`strat_fight`),
  `stratagem_engine.apply()` **сразу списывает CP** (`core/engine/phases/stratagem_engine.py:34-37`)
  и кладёт эффект `{"effect_id":"command_reroll","reroll_roll":...,"consumed":False}`
  (`:68-84`).
- Эффект «срабатывает» при совпадающем броске в резолве боя: `core/envs/warhamEnv.py:2407-2415`
  (shooting hit/wound) и `:2427-2450` (прочие фазы), там `rec["consumed"]=True` и
  `self._cmd_reroll_fired += 1`.
- **Реактивный механизм уже есть** (для overwatch/go_to_ground): `reaction_policy`
  (`warhamEnv.py:4304`), `_pending_reaction_trigger`, `_simulate_reaction_branch` (`:4355`),
  value-gate `make_reaction_value_policy`. Решение apply/pass принимается **в момент выстрела**.
- Waste = CP списан на arm, эффект не consumed к концу фазы/хода.

## Целевой дизайн (подход A, без ломки action-контракта)

Реактивность достигается **переносом решения и оплаты на момент броска**, а не переделкой
action-пространства (головы `strat_*` оставляем → **переобучение не требуется**):

1. **Arm без оплаты.** Выбор `command_reroll` в стратагем-окне всё ещё «армирует» эффект для
   юнита/типа броска, но **CP при arm НЕ списывается** (эффект помечается «armed, не оплачен»).
2. **Решение в момент броска (реактивно).** В точках consume (`warhamEnv.py:2407-2450`), когда
   приходит совпадающий бросок и у юнита есть armed-эффект: вместо «всегда применить» —
   прогнать через `reaction_policy` (apply/pass), как overwatch. Решение видит фактический бросок.
3. **Pay-on-apply.** Только при apply: списать CP (через единую точку CP-расхода), перебросить,
   `consumed=True`, `_cmd_reroll_fired += 1`. При pass / отсутствии броска — **CP не списывается**.
4. **Конец фазы/хода:** неоплаченные armed-эффекты просто отбрасываются (CP не теряется) → waste≈0.

### Затрагиваемые области (для декомпозиции GPT)
- `stratagem_engine.apply()` — НЕ списывать CP для command_reroll при arm; пометить эффект «unpaid».
- `warhamEnv` consume-точки (`2407-2450`) — вставить reaction-решение + pay-on-apply + CP-расход.
- CP-расход — единая функция (сейчас инлайн в stratagem_engine); для reroll вызывать на consume.
- Реакционная интеграция — собрать ctx для `reaction_policy` (как у overwatch: side, stratagem_id,
  phase, chosen, resolve_trigger) на reroll-решение.
- Телеметрия (`stratagem_trace`): `cmd_reroll_wasted` теперь ≈0 (arm бесплатен); `fired` = реальные
  рероллы. Уточнить смысл метрик в логе (waste→«armed-not-fired, CP не потрачен»).
- Гейт «нет цели» (`option_generator.command_reroll_options_for_unit`) — оставить (он про arm-опции).

## Инварианты (НЕ нарушить — движок!)
- Детерминизм/snapshot-restore: reaction-решение для reroll использует тот же snapshot/sim путь,
  что overwatch (`_simulate_reaction_branch`), без side-effects под симуляцией.
- CP никогда не уходит в минус; нельзя зарероллить без CP на момент apply.
- Один armed-эффект = максимум один реролл (consume один раз).
- Legacy-путь (нет `reaction_policy`): поведение «применять реролл по приходу броска» сохраняется,
  но **с pay-on-apply** (CP на consume, не на arm) — waste убирается даже без value-гейта.
- Action-контракт (`ordered_action_keys`, размер obs) НЕ меняется.

## Тестирование (TDD; движок обязателен)
- arm не списывает CP (modelCP неизменен после выбора command_reroll до броска).
- при совпадающем броске + reaction_policy=apply → CP списан один раз, реролл применён, fired+1.
- reaction_policy=pass → CP не списан, реролл не применён, waste без потери CP.
- нет совпадающего броска до конца фазы → CP не списан (armed отброшен).
- нет CP на момент apply → реролл не применяется (не уходит в минус).
- legacy (reaction_policy=None) → реролл по приходу броска, CP на consume.
- регресс: вся `tests/engine/phases` + `tests/engine` (сверка с baseline, не добавить падений);
  существующие стратагемные/CP тесты.
