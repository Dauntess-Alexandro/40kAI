# Стратагемы как действия — Под-проект 1: кодирование в action-space — Design

**Дата:** 2026-06-22
**Статус:** согласован, готов к плану реализации
**Большой проект:** «Стратагемы как класс действий в общем action-contract» (чтобы все RL-агенты — DQN/PPO/AZ/GAZ/GMZ/SMZ — учились выбирать стратагемы политикой/поиском). Это **под-проект 1 из 5**: 1) кодирование (этот док) → 2) маски легальности → 3) применение в env.step → 4) головы/priors по алго → 5) миграция/переобучение + eval/GUI.
**Происхождение:** вырос из «AZ должен по-настоящему решать Command Re-roll» — оказалось, что для настоящего поиска стратагему надо вносить в action-space; решено сделать это обобщённо для всех стратагем (ROI оправдан только так).

## Проблема

Сейчас стратагемы НЕ являются действиями политики (кроме частного `use_cp`/`cp_on` для Insane Bravery в command-фазе). Реакции (overwatch/go to ground) и fight-стратагемы (hungry_void/command_reroll) гонятся вне action-space — эвристикой/MC/MCTS-кандидатами вне joint_tuple. Поэтому ни один агент не «учится» выбирать стратагемы как часть действия. Цель под-проекта 1: дать контракту действия **пофазные головы выбора стратагемы**, заменив плоские `use_cp`/`cp_on`.

## Объём этого под-проекта (только форма контракта)

В области:
- Форма action-contract: новые головы, индексация стратагем, сборка/разбор `action_dict`.
- Один хелпер-источник истины: список стратагем (с подтипами) на фазу.
- `default_action_dict`, `action_tensor_to_dict`, `ordered_action_keys`, `action_sizes_from_env` — учитывают новые головы.

ВНЕ области (следующие под-проекты): маски легальности (2), интерпретация в `env.step`/применение (3), головы/priors/лоссы по алго и obs-изменения (4), переобучение/eval/GUI/checkpoint-миграция (5). Здесь действия пока **не применяются** — только кодируются и корректно сериализуются/десериализуются.

## Архитектура

### Фазы со стратагемами
`STRATAGEM_PHASES = (command, movement, shooting, charge, fight)` — фазы, где есть хотя бы одна стратагема в `REGISTRY` (или зарезервированы под будущие). Источник — `REGISTRY` из `core/engine/phases/stratagems.py`.

### Хелпер индексации (единая точка истины)
`stratagem_action_choices(phase) -> list[str]`:
- индекс 0 = `"none"`;
- далее — стратагемы, легальные в фазе (`d.phases` содержит phase), с **развёрткой подтипов**: `command_reroll` → `command_reroll:hit`, `command_reroll:wound` (в fight/shooting), `command_reroll:charge` (charge), `command_reroll:advance` (movement) — по тому же набору, что движок уже принимает в `reroll_roll`; прочие стратагемы — как один id (`hungry_void`, `insane_bravery`, ...).
- Порядок детерминирован (по порядку `REGISTRY` + фиксированный порядок подтипов), чтобы индекс был стабилен между контрактом, масками и применением.

`stratagem_choice_index(phase, choice_str) -> int` и обратное `stratagem_choice_str(phase, idx) -> str` — для сборки/разбора.

### Новые головы контракта (АДДИТИВНО)
В `ordered_action_keys` (после per-unit голов) для каждой фазы из `STRATAGEM_PHASES`:
- `strat_<phase>` ∈ `{0 .. len(stratagem_action_choices(phase))-1}` — выбор стратагемы (0 = none).
- `strat_<phase>_unit` ∈ `{0 .. len_model-1}` — целевой юнит.

`BASE_ACTION_HEADS`: **`use_cp`/`cp_on` СОХРАНЯЕМ** в этом под-проекте (они продолжают гнать Insane Bravery, движок не ломаем). Новые `strat_<phase>` головы добавляются **аддитивно** и пока **не применяются** (no-op до под-проекта 3). Удаление `use_cp`/`cp_on` и перенос Insane Bravery в `strat_command` — в под-проекте 3 (применение), когда новые головы начнут реально применяться. Так движок остаётся рабочим на всех шагах миграции.

### Один-на-фазу (YAGNI первого среза)
Максимум **одна стратагема на фазу на сторону** за ход. Контракт расширяем (добавить головы для стека позже). Документируется как осознанное ограничение.

### Сериализация
- `default_action_dict(len_model)`: все `strat_<phase>`=0, `strat_<phase>_unit`=0.
- `action_tensor_to_dict` / `action_sizes_from_env` — работают автоматически от `ordered_action_keys` (уже параметризованы), нужно лишь чтобы `env.action_space` объявлял новые ключи (правка пространства действий env — в этом под-проекте, т.к. это часть «формы контракта»; но интерпретация значений — под-проект 3).

## Тестирование (TDD)
- `stratagem_action_choices(phase)`: для fight/shooting содержит `command_reroll:hit/wound` + `none`; для charge — `command_reroll:charge`; command — `insane_bravery`; индекс 0 = none; детерминированный порядок.
- round-trip `stratagem_choice_index`/`_str`.
- `ordered_action_keys` содержит `strat_<phase>`/`strat_<phase>_unit` для всех `STRATAGEM_PHASES`, и больше НЕ содержит `use_cp`/`cp_on`.
- `action_sizes_from_env`/`action_tensor_to_dict`: новые головы корректных размеров; round-trip tensor↔dict.
- `default_action_dict`: новые ключи = 0.
- env.action_space объявляет новые ключи с верными `n`.

## Риски / open questions
- **Контракт ломает чекпойнты** — это ожидаемо и обрабатывается в под-проекте 5 (переобучение). В этом под-проекте — только убедиться, что `describe_obs_dim_mismatch`/контракт-сверки дают понятную ошибку (не молчат).
- **Удаление `use_cp`/`cp_on`** — найти ВСЕ потребители (legacy_patch insane_bravery в `legal_stratagem_options`, windowed_selfplay `make_command_decide_from_action_dict`, GUI/eval, тесты) и обновить под `strat_command`. Это в области (часть «формы»), но интерпретация-применение — под-проект 3. Здесь: контракт-уровень + заглушка применения может временно сохранить старое поведение Insane Bravery, чтобы не сломать движок до под-проекта 3. **Open:** делать ли мост `use_cp→strat_command` в этом под-проекте или принять временную «дыру» в применении Insane Bravery до п.3. Решить в начале плана.
- Размер action-space заметно растёт (≈2×5 голов) → больше параметров голов (под-проект 4).

## Критерии готовности
- Контракт содержит пофазные головы стратагем; хелпер индексации — единая точка истины; round-trip сериализации зелёный; `use_cp`/`cp_on` убраны из контракта (или явно смощены); регрессии контракт-тестов учтены.
- Применение значений НЕ требуется (под-проект 3) — но движок не должен падать на новом контракте (Insane Bravery либо смощен, либо явно помечен «применяется в п.3»).
