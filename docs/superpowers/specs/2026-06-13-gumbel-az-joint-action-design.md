# Gumbel AlphaZero — Joint-action (координатный) режим — Design Spec

**Дата:** 2026-06-13
**Статус:** утверждён пользователем (брейншторм), готов к написанию плана реализации.
**Базовый режим:** `gumbel_az` (см. `2026-06-13-gumbel-alphazero-design.md`). Это — опциональный апгрейд
поиска «Вариант B (joint-action)», заложенный в исходном спеке как будущий.

## Цель

Добавить в `gumbel_az` опциональный **joint-action** поиск, который учитывает взаимосвязь между головами
действия (главное — «движение ↔ видимость/стрельба»), за **гиперпараметром-тумблером** `joint_action`
(0/1, **по умолчанию 0** = текущий per-head). Поле — **первым** в группе «Поиск» вкладки GUI.

## Решение брейншторма (что и почему)

- **Реализация — координатная (coordinate ascent), а НЕ полный совместный top-m.** Причины:
  1. **Контракт обучения.** AZ-learner учит **по-головные** policy-таргеты. Координатный режим их даёт
     естественно (каждая голова по-прежнему делает свой SH → своё чистое распределение). Полный совместный
     потребовал бы восстанавливать по-головные таргеты из малой выборки комбинаций → шумные таргеты.
  2. **Стоимость.** Координатный ≈ как сейчас (тот же бюджет роллаутов, меняется только контекст соседних
     голов). Полный совместный — кратно дороже (что противоречит только что введённому `simulate_enemy=0`
     ради практичности).
  3. **Покрытие.** 8 голов → тысячи комбинаций; при малом `m` полный совместный покрывает пространство
     слабо. Координатный детерминированно проходит головы с учётом уже выбранных и стабильно ловит главную
     связку.
  4. **Риск.** Координатный — ~2 строки в цикле `run()`. Полный — переписывание ядра + новый путь деривации
     таргетов.
- **Честные ограничения (записаны явно, не обещаем чуда):**
  - На **eval/play/Viewer бот играет greedy** (без поиска). Значит joint-action влияет **только на self-play
    при обучении**; выигрыш — **косвенный**: когерентнее self-play + чище таргеты → лучше выученная сеть →
    лучше greedy-игра. Прямого улучшения хода «в бою» сейчас нет.
  - Зависит от **порядка голов** (один проход: ранние головы не видят поздние). Порядок у нас уже удачный —
    `move` раньше `shoot`, так что ключевая связка ловится.
  - Потолок держат **depth-1** и **качество value-сети**. Ожидаемый эффект — **единицы процентов winrate**,
    не «в разы». Поэтому режим — **измеримый тумблер**, а не дефолт.

## Алгоритм (единственное изменение в ядре)

В `GumbelAlphaZeroSearch.run()` цикл по головам уже считает `base_action` (greedy-prior argmax по каждой
голове) и ищет каждую голову в этом контексте. Изменение:

- Головы обрабатываются в существующем порядке (`ordered_action_keys` — он уже ставит `move` раньше `shoot`).
- При `joint_action=1`: **после** выбора финального действия головы `h` (включая дебютную стохастику)
  немедленно пишем его в `base_action[h]`, чтобы **следующие** головы искали уже в этом контексте.
- При `joint_action=0`: `base_action` остаётся greedy-prior на весь ход (текущее поведение, без изменений).

Что **НЕ меняется**: контракт `run()` (`(policy_targets, selected_actions, value)`), деривация по-головных
policy-таргетов (каждая голова — свой completed-Q `pi`), стоимость роллаутов, детерминизм при сиде,
фолбэки (нет легальных / NaN-приоры). `selected_actions[h]` = финальное действие головы `h` (как сейчас).

> v1 — один проход в естественном порядке голов. Второй проход (полная взаимная согласованность) и
> позиционно-первый порядок — возможные будущие уточнения, **вне scope**.

## Конфиг и обвязка

- **`GumbelAZSearchConfig`** (`core/models/gumbel_alphazero_search.py`): новое поле `joint_action: bool = False`.
- **`train.py`**: `GAZ_JOINT_ACTION` (env-оверрайд + `GAZ_CFG.get("joint_action", 0)`); добавить ключ в
  `_gaz_cfg_payload()`; `_build_az_search` прокидывает `joint_action` в `GumbelAZSearchConfig`.
- **`hyperparams.json`** → секция `gumbel_az`: `"joint_action": 0`.
- **`app/gui_qt/gaz_hyperparams_defaults.py`**: `joint_action: 0` в `_GAZ_BASE`; ключ в `GAZ_HYPERPARAM_KEYS`;
  **первым** в `keys` группы `search` у `GAZ_GROUPS`; тултип; **первым** в `GAZ_BASIC_KEYS` (чтобы тумблер
  был виден и в «Базовом наборе»).
- **`app/gui_qt/main.py`** `_start_training` (ветка `gumbel_az`): прокинуть `GAZ_JOINT_ACTION`.

> Тумблер 0/1. Пресеты (fast/balanced/heavy) `joint_action` **не трогают** (оставляем 0 во всех — это
> ортогональная опция; пользователь включает осознанно).

## GUI-формулировка

- Ключ конфигурации: `joint_action`. Тултип: «0 (дефолт) = головы выбираются независимо (per-head). 1 =
  координатный режим: каждая следующая голова видит уже выбранные действия предыдущих (движение→стрельба→…),
  ловит связку «куда встал ↔ кого стреляешь». Стоимость ≈ как при 0. Работает в self-play обучения (на eval
  игра greedy)».
- Размещение: **первый** ключ группы «Поиск» (группа и так первая на вкладке Gumbel AlphaZero).

## Тесты

- **Юнит** (`tests/models/test_gumbel_alphazero_search.py`):
  - `joint_action=0` ≡ текущее поведение (регрессия: те же `selected_actions`/таргеты на стаб-сети+env).
  - На сконструированном fake-env, где итог зависит от **согласованности** двух голов (напр. value выше,
    когда выбор поздней головы совпадает с выбором ранней), `joint_action=1` выбирает согласованную пару, а
    `joint_action=0` — нет.
  - При обоих значениях: по-головные таргеты валидны (сумма=1, масса только на легальных), детерминизм при
    сиде, фолбэк для головы без легальных.
- **Интеграция** (`tests/engine/test_gumbel_az_integration.py`): эпизод с `joint_action=1` на mini-env —
  валидные транзишены, env восстановлен.
- **GUI** (`tests/gui_qt/test_gaz_hyperparams_load.py`): `joint_action` грузится/коэрсится; присутствует в
  `GAZ_HYPERPARAM_KEYS` и первым в группе `search`.

## Критерии готовности

1. **Функционал:** тумблер `joint_action` виден первым в «Поиск»; `0` — без изменений (регрессия зелёная);
   `1` — train идёт, эпизоды завершаются, метрики считаются; новые тесты зелёные.
2. **Логи:** в `[GAZ][CONFIG]` видно `joint_action=0/1` для триажа.
3. **Качество (A/B) — отдельным шагом, не блокер v1:** через `eval.py` сравнить winrate моделей, обученных с
   `joint_action=0` и `=1` при равном бюджете. Если профита нет — режим остаётся выключенным (это ожидаемый и
   приемлемый исход; цель v1 — корректный измеримый тумблер без регрессий).

## Вне scope

- Полный совместный top-m (Gumbel по комбинациям) и деривация таргетов из выборки комбинаций.
- Второй проход координатной оптимизации и настраиваемый порядок голов.
- Gumbel-поиск (в т.ч. joint) на инференсе eval/play/Viewer — там по-прежнему greedy.
- Изменение пресетов под `joint_action` (остаётся 0 во всех).
