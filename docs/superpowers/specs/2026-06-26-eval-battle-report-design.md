# Eval Battle Report — дизайн

- **Дата:** 2026-06-26
- **Статус:** утверждён (дизайн), готов к плану реализации
- **Область:** `eval.py` (eval-прогон), новый модуль `core/eval/battle_report.py`, GUI-вкладка «Оценка» (переиспользование тумблера)

## 1. Контекст и мотивация

Сейчас «Детальный лог» eval (`EVAL_ACTION_TRACE`) пишет плоские `[TRACE]`/`[WH40K]` строки в
`runtime/logs/LOGS_FOR_AGENTS_EVAL.md`. Этого достаточно для отладки, но неудобно как «отчёт о
партии»: тяжело читать человеку, нет цельной картины «кто как ходил» по обеим сторонам, а reward
в логе — односторонний (только learner/model; `env.step()` считает шейпинг строго со стороны model,
`enemyTurn` награду не считает).

Нужен **battle report** — цельное, красивое и понятное описание партии, читаемое и человеком, и ИИ
(для последующего анализа), с результатом по **обеим сторонам**.

## 2. Принятые решения (из брейншторма)

1. **«Результат двух сторон» = per-side scoreboard**, НЕ RL-reward врага. Полную симметричную награду
   врага НЕ делаем (это крупная правка наград движка + риск регрессий). В отчёт идут объективные
   per-side метрики из `env.info`.
2. **reward-строку в scoreboard НЕ включаем вообще** (ни P1, ни оценку P2). Реальный reward learner
   остаётся в обычном логе/метриках. Это убирает любую правку наград движка из объёма.
3. **Формат — markdown: нарратив + таблицы.** Markdown структурирован сам по себе → читается и
   человеком, и ИИ (заголовки/таблицы). Отдельного JSON-приложения не делаем.
4. **Охват — отдельный файл на каждую партию** + сводный `index.md`.
5. **Включение — переиспользуем тумблер «Детальный лог»** (`EVAL_ACTION_TRACE`). Включён детальный
   лог → дополнительно генерируется battle report. Отдельного тумблера не вводим.
6. **Нарратив — только значимые действия.** Пустые/ничего-не-делающие фазы сворачиваем (не шумим
   строками «ничего не сделал»).

## 3. Архитектура: collector → renderer → writer

Три изолированные части (выбран подход A из трёх рассмотренных — отдельный сборщик данных + чистый
рендер + писатель файлов; альтернативы «рендер прямо в run_episode» и «парсинг своих лог-строк»
отклонены как нетестируемые/хрупкие):

Новый модуль `core/eval/battle_report.py`:

- **`BattleReportCollector`** — накапливает структурированную запись партии. Методы-хуки вызываются
  из `run_episode` в тех же точках, что и трейс:
  - `start_game(meta)` — шапка: индекс игры, learner_side/opponent_side (P1/P2), фракции, agent_id/algo
    обеих сторон, миссия, seed, назначение сторон (вкл. swap).
  - `record_side_phase(battle_round, side, action_dict, units, info_before, info_after)` — пофазные
    значимые действия стороны (move/shoot/charge/fight) с именами юнитов и целями.
  - `record_scoreboard(battle_round, info)` — снапшот scoreboard на конец раунда.
  - `finish_game(winner, end_reason, length, final_info)` — финал.
  - `to_record() -> BattleRecord` — неизменяемая структура для рендера.
- **`render_battle_report(record: BattleRecord) -> str`** — ЧИСТАЯ функция record→markdown. Без I/O.
  Главная точка тестирования.
- **`write_battle_report(run_dir, game_idx, markdown) -> Path`** и **`update_index(run_dir, summary_row)`**
  — запись `game_NNN.md` и обновление `index.md`.

Сбор данных НЕ зависит от текстового трейса (свои структуры), но гейтится тем же флагом
`trace_enabled` (тумблер «Детальный лог»).

## 4. Модель данных (`BattleRecord`)

- **header:** `game_idx`, `learner_side`, `opponent_side`, `p1_faction`, `p2_faction`,
  `learner_agent` (id, algo), `opponent_agent` (id/algo или `heuristic`), `mission`, `seed`,
  `assignment_label` (A/B при swap).
- **rounds:** список по battle round; каждый round:
  - `per_side` → для `model` и `enemy`: список значимых фаз (`phase`, человекочитаемая строка с
    юнитом/целью, опц. `damage`).
  - `scoreboard` — снапшот на конец раунда (см. §6).
- **final:** `winner` (P1/P2/draw), `end_reason`, `length` (ходов), `final_scoreboard`.

Производные величины:
- **урон стороны** = суммарная потеря HP противника между снапшотами (накопительно).
- **потери стороны** = число своих юнитов, ушедших в 0 HP.

Источники: `info["model health"]`/`info["player health"]` (списки HP), `info["model VP"]`/
`info["player VP"]`, `info["model controlled objectives"]`/`info["player controlled objectives"]`,
`info["battle round"]`, `info["turn"]`, `info["winner"]`, `info["end reason"]`; имена юнитов —
`model_units[i].showUnitData()["Name"]` / `enemy_units[i].showUnitData()["Name"]`.

## 5. Формат отчёта (RU, markdown)

```markdown
# Battle Report — Партия 7
P1 Space Marines (learner, dqn)  vs  P2 Orks (opponent, ppo) · миссия Only War · seed 42

## Battle Round 2
### P1 (Space Marines)
- Движение: Intercessors → точка B
- Стрельба: Intercessors → Ork Boyz: −4 HP
### P2 (Orks)
- Движение: Boyz → к центру
- Бой: Boyz × Intercessors: −3 HP

| Сторона | VP | HP    | контроль | урон (накопл.) | потери |
|---------|----|-------|----------|----------------|--------|
| P1      | 5  | 18/30 | 2        | 22             | 1      |
| P2      | 3  | 11/30 | 1        | 17             | 3      |

## Итог (After-Action)
Победитель: P1 · причина: VP-margin на turn-limit · ходов: 24
(итоговый scoreboard — та же таблица)
```

Пустые фазы у стороны не выводятся; если сторона за раунд ничего значимого не сделала — строка
«— без активных действий —».

## 6. Scoreboard — колонки

`Сторона | VP | HP (текущ/макс) | контроль точек | урон (накопл.) | потери (юнитов)`.
Reward-колонки нет (решение §2.2).

## 7. Интеграция

- **`eval.py run_episode`:** в хук-точках `_model_half` / `_enemy_half` / step-result добавляются
  однострочные вызовы collector'а под `if trace_enabled`. В конце эпизода: `render_battle_report` +
  `write_battle_report`. Логика боя/трейса не меняется.
- **Последовательный путь** (`main`): пишет файлы сам, обновляет `index.md` по ходу.
- **Параллельные воркеры** (`eval_worker_entry`/`_build_eval_runtime_for_worker`): каждый воркер
  собирает свой `BattleRecord`, рендерит и пишет свой `game_NNN.md` (idx-based — без коллизий).
  Главный процесс собирает `index.md` после агрегации результатов.

## 8. Файлы и пути

- `artifacts/results/battle_reports/<run_id>/game_NNN.md` — отчёт партии.
- `artifacts/results/battle_reports/<run_id>/index.md` — таблица: партия, P1 фракция, P2 фракция,
  winner, VP P1/P2, ссылка на файл.
- `<run_id>` — идентификатор прогона eval (тот же, что у метрик/результатов).
- Не трогаем guarded `runtime/logs/LOGS_FOR_AGENTS_EVAL.md`.

## 9. Включение

Переиспользуем `EVAL_ACTION_TRACE` (GUI «Детальный лог»). Отдельного env/тумблера нет. Когда выключен
— отчёт не генерируется, накладных расходов нет.

## 10. Тестирование

- **Unit `render_battle_report`** (главный): синтетический `BattleRecord` → проверяем наличие шапки,
  секций по раундам, обеих сторон, scoreboard-таблицы, блока «Итог», и что пустые фазы свёрнуты.
- **Unit `BattleReportCollector`**: подаём последовательность фаз/снапшотов → корректная запись,
  верный расчёт урона/потерь из дельт HP.
- **Smoke (лёгкий)**: короткий eval-эпизод с включённым тумблером → файл создан, содержит шапку и
  итог. Держать минимальным (или пропустить, если дорого по ОЗУ/времени).

## 11. Вне объёма (YAGNI)

- Симметричная RL-награда врага и любые правки наград движка.
- JSON-приложение / машинный блок (выбран нарратив+таблицы).
- Стриминг отчёта в реальном времени.
- Картинки/диаграммы поля (только текст/markdown).

## 12. Открытые риски

- **Параллельный `index.md`:** во избежание гонок индекс собирает только главный процесс после
  агрегации; воркеры пишут лишь свои `game_NNN.md`.
- **Большой объём при сотнях игр:** отдельные файлы на игру это сглаживают (главный лог не растёт);
  при необходимости позже добавить ретеншн/лимит — вне текущего объёма.
