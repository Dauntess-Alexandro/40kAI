# Стратагемы как действия — Под-проект 2: маски легальности — Design

**Дата:** 2026-06-22
**Статус:** согласован, готов к плану реализации
**Большой проект:** «Стратагемы как класс действий». **Под-проект 2 из 5**: 1) кодирование (готово, `53e9d50b`/реализация `e3f1fab4..2f37e214`) → **2) маски (этот док)** → 3) применение → 4) головы/priors алго → 5) переобучение/eval/GUI.

## Проблема

Под-проект 1 добавил в контракт головы `strat_<phase>` / `strat_<phase>_unit` (по фазам), но **масок легальности для них нет** — `build_action_masks_by_head`/`get_legal_action_masks_by_head` их не заполняют. Без масок политика/MCTS не знает, какие выборы стратагемы легальны (хватает ли CP, не исчерпан ли usage_limit, есть ли юнит с keyword, жив ли юнит). Цель под-проекта 2: дать факторизованные лояльные маски новым головам — **зеркало существующего `use_cp`/`cp_on`**.

## Объём (только маски)

В области: маски `strat_<phase>` и `strat_<phase>_unit` в `build_action_masks_by_head` (и согласованно в `get_legal_action_masks_by_head`, если они расходятся). Хелпер легальности «применима ли стратагема сейчас» (CP/usage_limit/keyword/alive).

ВНЕ области: применение значений голов (под-проект 3 — здесь головы остаются no-op), priors/лоссы алго (4), переобучение/GUI (5). Маски лишь объявляют легальность; ничего не применяется.

## Архитектура (факторизовано + лояльно, как use_cp/cp_on)

Прецедент: [warhamEnv.py:1819-1841](../../../core/envs/warhamEnv.py#L1819-L1841) — факторизованные маски, индекс 0 всегда легален, fallback на [0], невалидные комбо = no-op при применении.

### `strat_<phase>` (выбор стратагемы)
- Индекс 0 (`none`) — всегда `True`.
- Индекс k (категория из `stratagem_action_choices(phase)[k]`) — `True` ⟺ стратагема применима стороной сейчас:
  - `cp >= by_id(base_id).cp_cost` (где `base_id` — id без подтипа: `"command_reroll:hit"` → `"command_reroll"`);
  - НЕ `usage_limit_reached(env, side, by_id(base_id), phase=<phase>.value)`;
  - есть ≥1 живой юнит стороны с нужным `keyword_req` (если `keyword_req` пуст — достаточно ≥1 живого юнита).
- **Без жёсткой привязки к `env.phase`** (action_dict — на весь ход; гейтинг по фазе сломал бы предвыбор). Осознанное отличие от `use_cp` (тот легален только в command-фазе).
- Подтипы (`command_reroll:hit/wound/charge/advance`) маскируются по тем же критериям (легальность от подтипа не зависит — он лишь параметр броска).
- Fallback: если ни один k не легален → только [0]=True.

### `strat_<phase>_unit` (юнит)
- Живые юниты стороны → `True` (лояльно, как `cp_on`). Keyword-связка «стратагема X только на юните Y» не выразима факторизованно → валидируется при применении (под-проект 3).
- Fallback: нет живых (край) → [0]=True.

### Хелпер легальности
`_stratagem_choice_legal(side, phase, choice_str) -> bool` (или встроенный цикл) — единая логика «применима ли категория сейчас», переиспользует `by_id`, `usage_limit_reached`, `_unit_has_keyword`, CP стороны. Источник категорий — `stratagem_action_choices(phase)` (под-проект 1).

### Совместимость
- `use_cp`/`cp_on` маски — **без изменений** (под-проект 1 их сохранил; применение Insane Bravery работает по-старому).
- Взаимодействие новых голов с `reaction_policy` (подавлять ли, как `use_cp[0]`-only при rvp_on) — **НЕ в этом под-проекте**; решается в под-проекте 3 (применение). Здесь маскируем строго по легальности.

## Тестирование (TDD)
- `strat_<phase>` маска: [0] всегда True; при CP=0 — только [0]; при CP≥cost и наличии keyword-юнита — соответствующий индекс True.
- usage_limit: применённая PER_PHASE/PER_BATTLE стратагема (запись в `stratagem_used`) → её индекс False.
- keyword: hungry_void (keyword necrons) — индекс True только если есть живой NECRONS-юнит; иначе False; command_reroll (без keyword) — True при CP.
- Подтипы command_reroll (hit/wound) — оба True/False синхронно (легальность одинаковая).
- `strat_<phase>_unit`: живые юниты True, мёртвые False; все мертвы → [0]=True.
- Размерность маски каждой головы == размеру соответствующего space (`stratagem_action_choices(phase)` / len units).
- Маски совместимы с `FactorizedLegalMasks.as_ordered_list` / потреблением в DQN/PPO/AZ (все ordered-ключи имеют маску — иначе KeyError).
- Регрессия: `use_cp`/`cp_on` маски не изменились; baseline `tests/engine/` 23.

## Риски / open questions
- **Полнота масок:** все потребители `get_legal_action_masks_by_head` ожидают маску для КАЖДОГО ordered-ключа. Новые ключи добавлены в контракт (под-проект 1) → если маски для них не заполнить, потребители (`_masks_for_side`, `as_ordered_list`, reaction-value-policy) упадут KeyError. Поэтому под-проект 2 **обязателен** для консистентности (под-проект 1 оставил их без масок — проверить, не падает ли что-то уже сейчас; если падает — это и чиним здесь).
- **env.phase-агностичность:** маска не гейтит по текущей фазе. Это значит политика может «выбрать» стратагему для фазы, которая в этом ходу не наступит для юнита (напр. реролл стрельбы у юнита без целей) — применение (под-проект 3) сделает это no-op. Приемлемо (лояльность).
- **Open:** где именно `get_legal_action_masks_by_head` vs `build_action_masks_by_head` (два пути?) — в начале плана свериться, что маски заполняются в обоих/общем месте.

## Критерии готовности
- Все `strat_<phase>`/`strat_<phase>_unit` головы получают корректные факторизованные маски; index 0 безопасен; legality по CP/usage/keyword/alive.
- Потребители масок не падают (все ordered-ключи покрыты); `use_cp`/`cp_on` не тронуты; регрессий нет (baseline 23).
- Применение по-прежнему не требуется (под-проект 3).
