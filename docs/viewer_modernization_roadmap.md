# 40kAI Viewer Modernization Roadmap (подробно)

## Цель документа

Этот roadmap описывает поэтапную модернизацию игрового вьювера `app/viewer` с фокусом на:

- более современный визуал;
- единый стиль с `app/gui_qt`;
- управляемые риски (без "переписать всё сразу");
- улучшение поддержки и скорости итераций команды.

Документ специально построен так, чтобы его можно было использовать как рабочий план на несколько спринтов.

**Исполнимый спринт-план (контракты, флаги, CI, упаковка):** [viewer_migration_plan.md](viewer_migration_plan.md).

---

## 1) Исходная точка (As-Is)

### 1.1 Архитектура сейчас

- Лаунчер/настройки/обучение: `app/gui_qt` (QML + PySide6).
- Игровой вьювер: `app/viewer` (PySide6 Widgets + `QOpenGLWidget` + `QPainter`-слои).
- Игровая логика и правила: `core/engine/*`.

### 1.2 Ключевые ограничения

- Вьювер сильно связан с `GameController`.
- В `opengl_view.py` объединены:
  - геометрия;
  - камера/трансформ;
  - рендер всех слоёв;
  - FX;
  - hit-test и интерактив.
- Визуальный язык `viewer` отличается от QML-лаунчера (цвета, панели, ритм UI).

### 1.3 Почему не "сразу C++"

- Для цели "сделать красивее" C++ почти не даёт value сам по себе.
- Главный выигрыш достигается через:
  - дизайн-систему;
  - пайплайн слоёв и FX;
  - частичную миграцию UI в QML;
  - локальные shader-based улучшения.

---

## 2) Цели и KPI (To-Be)

### 2.1 Визуальные цели

- Единый стиль лаунчер + viewer (токены цвета/типографики/скруглений/отступов).
- Более "живой" рендер поля:
  - аккуратные свечения;
  - плавные анимации движения и событий;
  - более читаемые оверлеи.

### 2.2 Технические цели

- Разделить `opengl_view.py` на модульные слои.
- Вынести presentation-логику UI из `ViewerWindow` в контроллер-свойства/сигналы.
- Подготовить базу под гибрид Widgets + QML.

### 2.3 KPI (измеримые)

- Время на внедрение визуальной фичи: минус 30-40%.
- Кол-во регрессий в интерактиве поля (click/select/hover): < 2 на релиз.
- Median FPS (на эталонной сцене): не хуже текущего, целевой рост +15% при FX.
- Воспринимаемое качество UI (внутренний опрос): +2 пункта по 10-балльной шкале.

---

## 3) Принципы внедрения

1. **Никакого big-bang rewrite**: только инкрементальные шаги.
2. **Совместимость прежде всего**: контракты состояния и событий сохраняются.
3. **Feature flags** для рискованных изменений.
4. **Сначала foundation, потом вау-эффекты**.
5. **Каждая фаза имеет Definition of Done**.

---

## 4) Общий график (предложение на 10-14 недель)

- Фаза 0: подготовка и baseline (1 неделя)
- Фаза 1: единая дизайн-система (1-2 недели)
- Фаза 2: модульный рендер-скелет (2-3 недели)
- Фаза 3: визуальные улучшения поля (2-3 недели)
- Фаза 4: гибридный UI (Widgets + QML) (2-3 недели)
- Фаза 5: selective shaders / polish / hardening (2 недели)

Ниже — подробно по каждой фазе.

---

## 5) Фаза 0 — Baseline и подготовка (1 неделя)

### 5.1 Задачи

- Зафиксировать текущий baseline:
  - FPS в типовых сценах;
  - загрузка CPU/GPU;
  - latency клика/hover;
  - скриншоты ключевых экранов.
- Сформировать "golden scenarios":
  - старт матча;
  - деплой;
  - движение отряда;
  - стрельба с FX;
  - долгий лог.
- Добавить "режим сравнения" в QA-плейбук (до/после).

### 5.2 Артефакты

- `docs/viewer_baseline_metrics.md`
- Папка эталонных скриншотов/видео (например, `docs/media/viewer_baseline/`).
- Чек-лист smoke regression.

### 5.3 Definition of Done

- Есть измеримый baseline.
- Есть повторяемые сценарии для сравнения каждого этапа.

---

## 6) Фаза 1 — Единый визуальный foundation (1-2 недели)

### 6.1 Цель

Сделать так, чтобы viewer и gui_qt визуально читались как один продукт.

### 6.2 Задачи (по приоритету)

1. Ввести единые токены темы:
   - цвета (bg/surface/text/accent/warn/danger);
   - типографика (размеры, веса, line-height);
   - радиусы/отступы/бордеры.
2. Подключить токены в `app/viewer/styles.py`.
3. Привести основные виджеты `ViewerWindow` к этим токенам:
   - `QGroupBox`, `QLabel`, `QTableWidget`, `QPushButton`.
4. Сверить иконки и масштаб:
   - единая библиотека ассетов;
   - согласованный размер по DPI.
5. Вынести настройки анимаций в конфиг (расширить `viewer_config.json`).

### 6.3 Быстрые win-фичи

- Мягкий hover/highlight для интерактивных зон.
- Подправить контраст текста/бордеров для читаемости.
- Убрать "перепады" стилей между панелями.

### 6.4 Риски

- Непредсказуемость внешнего вида на разных DPI.
- Неоднородность шрифтов между ОС.

### 6.5 Mitigation

- Тест на 100% / 125% / 150% scale.
- Fallback-шрифты в теме.

### 6.6 Definition of Done

- 80-90% UI viewer использует общие токены.
- На эталонных разрешениях нет деградации читаемости.

---

## 7) Фаза 2 — Разделение рендера на слои (2-3 недели)

### 7.1 Цель

Раздробить монолит `opengl_view.py`, чтобы визуал можно было развивать безопасно и быстро.

### 7.2 Предлагаемая целевая структура

```text
app/viewer/
  rendering/
    board_geometry.py
    board_camera.py
    layer_context.py
    layers/
      ground_layer.py
      grid_layer.py
      movement_layer.py
      objectives_layer.py
      units_layer.py
      shooting_layer.py
      fx_layer.py
      labels_layer.py
    hit_test.py
    animation_timeline.py
```

### 7.3 Пошаговый рефактор

1. Ввести `LayerContext` (painter, transform, state snapshot, config).
2. Экстрагировать сначала "простые" слои:
   - grid;
   - labels;
   - objectives.
3. Затем средние:
   - movement;
   - shooting overlays.
4. В конце сложные:
   - units;
   - FX/particles.
5. Оставить `OpenGLBoardWidget` как оркестратор:
   - подготовка контекста;
   - вызов layer stack;
   - финальный debug overlay.

### 7.4 Что не трогать в этой фазе

- Бизнес-логику `GameController`.
- Протоколы состояния.
- Поведение кликов/команд.

### 7.5 Definition of Done

- `opengl_view.py` заметно уменьшен.
- Каждый слой тестируется/валидируется отдельно.
- Визуально поведение совпадает с baseline (допустимы мелкие пиксельные отличия).

---

## 8) Фаза 3 — Улучшение поля и FX (2-3 недели)

### 8.1 Цель

Добавить "красоту", не ломая стабильность.

### 8.2 Направления

1. **Грид и фон**
   - менее "жёсткая" сетка;
   - major/minor линии;
   - лёгкая виньетка/глубина.
2. **Выделения**
   - единый language для selection/hover/target;
   - пульсация через configurable easing.
3. **Анимации движения**
   - усреднённые тайминги;
   - `smoothstep`/`cubic_out`/custom кривые;
   - stagger для моделей в отряде.
4. **Боевые FX**
   - аккуратный bloom-стиль glow;
   - унифицированные вспышки попаданий;
   - читаемые popups урона.

### 8.3 Технический подход

- Начать с улучшений внутри существующего QPainter-канала.
- Для дорогих эффектов — ограниченные offscreen passes.
- Включать через флаги в `viewer_config.json`:
  - `fx_quality = low|medium|high`
  - `postprocess = off|light`

### 8.4 Перф-контроль

- Мониторить FPS/frametime на каждой подзадаче.
- При падении >10% от baseline:
  - оптимизировать;
  - снижать качество по пресету;
  - откладывать эффект.

### 8.5 Definition of Done

- Визуал заметно современнее по эталонным сценариям.
- Нет критичных просадок на целевых машинах.

---

## 9) Фаза 4 — Гибридный UI: Widgets + QML (2-3 недели)

### 9.1 Цель

Перенести "хром" (панели/статусы/часть управления) в QML, сохранив стабильное поле.

### 9.2 Стратегия миграции

- Поле (`OpenGLBoardWidget`) остаётся в Widgets.
- Правую колонку постепенно переводим в QML:
  1) статус/очки;
  2) команды;
  3) лог.

### 9.3 Слой интеграции

- `ViewerController(QObject)`:
  - свойства для QML (round/phase/vp/cp/active unit/...);
  - сигналы обновлений;
  - слоты для действий UI.

### 9.4 Критически важно

- Не дублировать логику: QML только presentation.
- Все команды идут через один контроллер в Python.
- Ясная ownership-модель объектов (избегать dangling ссылок в QML context).

### 9.5 Definition of Done

- Ключевые панели viewer отрисовываются в QML.
- Поведение и команды эквивалентны текущим.
- Пользователь получает единый UX с `gui_qt`.

---

## 10) Фаза 5 — Selective shaders + hardening (2 недели)

### 10.1 Цель

Точечно добавить shader-based эффекты там, где QPainter уже не даёт нужного уровня.

### 10.2 Кандидаты на шейдеры

- Highlight platform (scanlines/noise glow).
- Энергетические трассеры выстрелов.
- Лёгкий compositing/bloom pass для выделений.

### 10.3 Ограничения

- Только узкие hotspots, без тотальной миграции рендера.
- Каждый шейдер имеет fallback.

### 10.4 Hardening

- Стресс-тест длинных сессий.
- Проверка утечек и деградации FPS со временем.
- Документация перф-профилей.

### 10.5 Definition of Done

- Топ-2/3 эффектов переведены в shader path.
- Fallback стабилен и покрыт smoke-тестами.

## 12) Роли и ответственность

- **Tech Lead / Архитектор**
  - принимает решения по фазам и контрактам;
  - утверждает performance budget.
- **UI/UX Engineer**
  - тема, токены, QML-компоненты, визуальное качество.
- **Rendering Engineer**
  - рефактор слоёв, FX pipeline, shader pilots.
- **Gameplay Integrator**
  - корректность интерактива (select/move/shoot).
- **QA**
  - regression matrix, скриншотные сравнения, стресс-тест.

Если команда маленькая, роли могут совмещаться, но ответственность должна оставаться явной.

---

## 13) Риски и управление ими

### 13.1 Технические риски

- Регрессии интерактива при разбиении слоёв.
- Перф-падение на старом железе.
- Усложнение гибридного Widgets+QML стека.

### 13.2 Организационные риски

- "Scope creep" (добавление фич раньше foundation).
- Отсутствие baseline и "споры по вкусу".

### 13.3 Mitigation

- Feature flags + этапные релизы.
- Обязательные compare-сценарии перед merge.
- Freeze на новые фичи в рефактор-фазах.

---

## 14) QA стратегия

### 14.1 Матрица тестов

- Разрешения: 1080p / 1440p / 4K.
- DPI: 100 / 125 / 150%.
- Перф: low-end / mid / high профили.
- Сценарии: deploy, movement, shooting, long log, reset game.

### 14.2 Типы тестирования

- Smoke (каждый PR).
- Visual regression (эталонные скриншоты).
- Interaction regression (клики/hover/selection).
- Soak test (длинный прогон с FX).

### 14.3 Критерии релиза

- Нет blocker-багов.
- FPS в budget.
- Нет критичных расхождений с логикой матча.

---

## 15) Feature flags (рекомендуемый набор)

- `viewer.theme.v2` — новые токены/стили.
- `viewer.render.layers_v2` — модульный рендер pipeline.
- `viewer.fx.v2` — улучшенные эффекты.
- `viewer.ui.qml_panels` — QML правой колонки.
- `viewer.render.shader_highlights` — shader path для highlight.

Flags позволяют катить изменения постепенно и быстро откатывать рискованные части.

---

## 16) План релизов и откат

### 16.1 Канареечный rollout

1. Внутренний nightly.
2. Ограниченный круг пользователей/разработчиков.
3. Широкое включение по умолчанию после 1-2 недель стабильности.

### 16.2 Откат

- Отключение флагов без полного rollback кода.
- Документированный "safe profile" (`fx_quality=low`, shader off).

---

## 17) Бэклог визуальных улучшений (после roadmap)

- Тематические окружения (несколько ground palettes).
- Расширенные kill/impact feedback эффекты.
- Cinematic camera moments (опционально).
- Минималистичный spectator HUD.

---

## 18) Минимально необходимый первый шаг (рекомендуется начать с этого)

Если нужно стартовать сразу и быстро показать результат:

1. Ввести единые токены темы.
2. Перекрасить viewer под стиль `gui_qt`.
3. Нормализовать шрифты/иконки/панели.
4. Добавить пресеты анимации в конфиг.

Это даст наиболее заметный визуальный эффект при минимальных рисках и без переписывания рендера.
