# План перехода к мультифракционному обучению по фазам

## Контекст
Сейчас обучение в основном идёт в симметрии (одинаковые фракции/юниты), поэтому self-play хорошо работает только в узком сценарии. При добавлении новой фракции с другими правилами нужна отдельная логика выбора стороны обучения и раздельного хранения моделей.

## Общий принцип
- Разделяем сущности: **сторона обучения (P1/P2)**, **фракция обучения**, **фракция противника**, **чекпойнт**.
- Сохраняем чекпойнты с явной метаинформацией, чтобы их можно было безопасно использовать в self-play/cross-play.
- Переходим от “одного snapshot” к пулу оппонентов и контролируемому матчмейкингу.

---

## Фаза 1 — Базовая разметка и сохранение (MVP)
**Цель:** начать сохранять модели в новой структуре, где явно указано кого обучали.

### Что делаем
1. Добавляем конфиг цели обучения через env:
   - `TRAIN_SIDE` (`P1`/`P2`)
   - `TRAIN_FACTION` (например, `Necron`)
   - `OPPONENT_FACTION` (например, `Necron`/`SpaceMarines`)
2. Формируем scope модели:
   - формат: `{TRAIN_SIDE}_{TRAIN_FACTION}_vs_{OPPONENT_FACTION}`
3. Складываем артефакты в директорию scope.
4. Сохраняем в checkpoint дополнительную мету:
   - `model_scope`
   - `training_target.side`
   - `training_target.train_faction`
   - `training_target.opponent_faction`
5. Пишем отдельный `checkpoint_meta.json` рядом с моделями, где хранится ссылка на последний актуальный checkpoint.

### Критерии готовности
- Можно запустить 2 тренировки с разными `TRAIN_SIDE` и получить изолированные папки моделей.
- В каждом checkpoint есть `training_target`.
- Есть `checkpoint_meta.json` с последним checkpoint.

### Статус
- [x] Реализовано (2026-03-22): базовая разметка цели обучения, новая структура сохранения и метаданные checkpoint.

## Фаза 2 — Выбор активной стороны обучения
**Цель:** обучать только выбранную сторону (P1 или P2), сохраняя вторую как оппонента.

### Что делаем
1. Явно закрепляем “trainable side” в train-loop.
2. Для противоположной стороны используем policy из пула/фиксированный checkpoint.
3. Добавляем в логи явную строку, какая сторона обучалась в эпизоде.

### Критерии готовности
- В одном запуске изменяются веса только у выбранной стороны.
- Логи однозначно показывают train side/opponent side.

## Фаза 3 — Opponent Pool (latest/best/random)
**Цель:** снизить переобучение на одного оппонента.

### Что делаем
1. Пул оппонентов из нескольких checkpoint.
2. Смесь sampling-стратегий: latest/best/random.
3. Простая статистика по каждому оппоненту (games/winrate/vp_diff).

### Критерии готовности
- Источник оппонента выбирается из пула, а не всегда один snapshot.
- В логах видно `opponent_source` и `opponent_id`.

### Статус
- [ ] Частично (зачатки уже есть в self-play, требуется привязка к scope/faction)

## Фаза 4 — Cross-play матрица между фракциями
**Цель:** корректно обучать при нескольких фракциях.

### Что делаем
1. Вводим match config: `(P1 faction/policy) vs (P2 faction/policy)`.
2. Балансируем sampling матчей между mirror и cross-faction.
3. Фиксируем benchmark-набор матчей для честной оценки.

### Критерии готовности
- Есть регулярная оценка на фиксированной матрице матчей.
- Чекпойнты продвигаются в “best” по метрике benchmark, а не по одному сценарию.

## Фаза 5 — Лига и стабильный прод
**Цель:** долгосрочная стабильность и отсутствие регрессий.

### Что делаем
1. Лига агентов: main, exploiter, historical.
2. Версионирование правил/миссий в метаданных.
3. Регрессионные тесты train/eval pipeline и валидация формата checkpoint.

### Критерии готовности
- Новая фракция добавляется без ручных “костылей” в сохранении моделей.
- При изменениях правил видно, какие checkpoint совместимы.

## Что запускать прямо сейчас (Phase 1)
Примеры:

```bash
TRAIN_SIDE=P1 TRAIN_FACTION=Necron OPPONENT_FACTION=Necron python train.py
TRAIN_SIDE=P2 TRAIN_FACTION=Necron OPPONENT_FACTION=Necron python train.py
```

Ожидаемо получатся отдельные scope-папки в `models/` и отдельные метаданные для каждого запуска.
