# Мульти-модельная работа над 40kAI

Как четыре модели работают вместе над проектом. Документ — памятка, чтобы не держать схему в голове.

## Роли (иерархия)

| Модель | Инструмент | Роль | Зона ответственности |
|--------|-----------|------|----------------------|
| **Claude Opus 4.8** | Claude Code | Главный архитектор | «ЧТО и ПОЧЕМУ»: подробный план, архитектура, вопросы заказчику, финальное ревью |
| **GPT 5.5 xhigh** | Roo Code (`orchestrator-40k`) | Дирижёр | «КАК»: ревью плана, декомпозиция, делегирование, ревью исполнителей до 100% |
| **GLM 5.2** | Roo Code | Исполнитель / дирижёр | Кодогенерация; 1M контекст / 131k вывод |
| **Kimi 2.7** | Roo Code | Исполнитель | Кодогенерация; 256k контекст / 32k вывод |

### Текущая расстановка (GPT вернулся)
- `orchestrator-40k` → **GPT 5.5 xhigh** (дирижёр) — разбирает план архитектора подробно, делегирует, ревьюит.
- `executor` → **GLM 5.2** (исполнитель).

Привязка моделей — в Roo UI (Settings → Modes → API Configuration на режиме). Смена — одна настройка, обратима.
GPT-дирижёру указано в `.roomodes`: **детально разбивать** план в bite-sized TDD-подзадачи перед делегированием.

## Два моста

```
┌──────── Claude Code ────────┐   мост 1: файлы в git   ┌──── Roo Code (одна сессия) ────┐
│ Opus — архитектор            │  (plans/*.md + ветки)   │ Дирижёр (Orchestrator)          │
│  • подробный план + вопросы  │  ⇄ ─────────────────── ⇄│  ├ new_task → executor-glm      │
│  • финальное ревью           │                          │  └ new_task → executor-kimi     │
└──────────────────────────────┘                          └─────────────────────────────────┘
                                          мост 2: внутри Roo, автоматический (Boomerang)
```

- **Мост 1 (Opus ⇄ дирижёр):** разные инструменты → только через файлы в git. Прямого канала нет.
- **Мост 2 (дирижёр ⇄ исполнители):** один инструмент (Roo) → встроенный режим Orchestrator делегирует через `new_task`, результат возвращается «бумерангом» автоматически.

## Цикл

```
1. Opus (Claude Code):  брейншторм → ПОДРОБНЫЙ план в plans/feature.md
                        (задачи, точные пути, сигнатуры, критерии приёмки, тесты)
                        + вопросы заказчику → коммит в main
        ↓ файл в git
2. Дирижёр (Roo):       читает план → ревьюит дизайн → дополняет/правит план в файле → коммит
        ↓ внутри Roo, авто
3. Дирижёр → исполнитель: под-задачи по очереди (new_task), результат бумерангом
        ↓
4. Дирижёр:             ревью КАЖДОГО результата, гоняет на доработку, пока не 100% + тесты зелёные
        ↓ файл/git (ветка готова)
5. Opus (Claude Code):  финальное ревью всего диффа (субагенты engine-regression-reviewer /
                        code-reviewer) → дополняет/дописывает → мерж
```

## Правила, без которых схема не работает

1. **Качество плана = потолок цепочки.** Исполнители тянут ровно настолько, насколько чётко нарезаны под-задачи. Главная работа Opus — исполнимый план: файлы, сигнатуры, тесты вперёд.
2. **Два уровня ревью — это фича.** Дирижёр ловит «код не по плану / баги» (построчно). Opus ловит «архитектура/инвариант движка поехали» (целостно). Не схлопывать.
3. **План коммитится в общую базу (main).** Roo-окно видит файл, только если он в его ветке. Планы и их правки фиксируются коммитом до передачи дальше.
4. **`AGENTS.md` = единые правила для всех.** Roo читает `AGENTS.md`, Claude — `CLAUDE.md` (он подключает `AGENTS.md`). Русский язык, Windows, правило коммитов, «отладка с логов» — действуют на все модели.
5. **Слабое звено — дирижёр.** На временной схеме (Kimi/GLM вместо GPT) дирижёр слабее → финальное ревью Opus обязано быть настоящим (диффы + субагенты), а не «доверяю дирижёру».

## Особые требования к Opus (главный архитектор)

- **Пиши ПОДРОБНЫЙ план** — детально, как для исполнителя послабее: явные шаги, точные пути к файлам, сигнатуры, критерии приёмки, тесты.
- **Уточняй вопросы у заказчика (пользователя) ПОБОЛЬШЕ** — не угадывай требования; перед финализацией плана задавай уточняющие вопросы.
- **Используй скиллы по максимуму** — `brainstorming` → `writing-plans` → `executing-plans`, `systematic-debugging`, `test-driven-development`, `requesting-code-review`/`verification-before-completion`, субагенты `engine-regression-reviewer`/`code-reviewer`. Процессные скиллы — раньше реализационных.
