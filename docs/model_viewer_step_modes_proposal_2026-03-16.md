# Предложение: пошаговый Viewer для модели (по ходам/фазам/юнитам)

## Цель
Сделать отображение действий **модели (AI)** в Viewer пошаговым на карте:
- по ходам,
- внутри хода по фазам,
- внутри фазы по юнитам.

Игрока (human side) оставляем как есть.

---

## Вариант 1 — Линейный «таймлайн событий» (рекомендуемый)

### Идея
Во время хода модели собирать последовательность элементарных событий и показывать их по кнопке `След. шаг`.

Пример событий:
- `turn_start`
- `phase_start(command)`
- `phase_start(movement)`
- `unit_action_start(unit=1, action=move)`
- `unit_action_end(unit=1, action=move, from=(x1,y1), to=(x2,y2))`
- `unit_action_start(unit=2, action=move)`
- ...
- `phase_start(shooting)`
- `unit_action_start(unit=1, action=shoot, target=enemy_2)`
- `unit_action_end(unit=1, action=shoot, result=hit/miss/damage)`

### UI
Минимальный блок рядом со статусом:
- `След. шаг` (главная кнопка)
- `Авто` (плей с задержкой)
- `Пауза`
- опционально `Назад`
- строка статуса: `Ход 1 • Фаза стрельбы • Юнит Intercessors #2`

### Что видит пользователь
- В командной фазе, если действий нет, просто переходит к следующему событию (без «пустого» визуального шага).
- В фазе движения: по одному юниту с обновлением позиции на карте.
- В фазе стрельбы: по одному юниту, подсветка атакующего + цели + результат.

### Плюсы
- Самая гибкая архитектура.
- Легко добавить фильтры (`только движение`, `только стрельба`) и экспорт replay.
- Можно переиспользовать и для train/eval разборов.

### Минусы
- Нужно аккуратно встроить сбор событий в текущий пайплайн действий.
- Потребуется единый формат event payload.

## Вариант 2 — «Пошаговые снимки состояния» (state snapshots)

### Идея
Сохранять состояние карты после каждого значимого действия модели.
`След. шаг` = показать следующий snapshot.

### Плюсы
- Простая отрисовка: просто подменяем состояние.
- Удобно для дебага (можно быстро сравнивать step N и N+1).

### Минусы
- Больше памяти (особенно при длинных матчах).
- Сложнее объяснить «что именно произошло» без отдельного event-слоя.

---

## Вариант 3 — Гибрид: события + чекпоинты фаз

### Идея
Храним события как в Варианте 1, но дополнительно делаем снапшот на границе фаз:
- старт/конец movement,
- старт/конец shooting,
- и т.д.

### Плюсы
- Хороший баланс: детальность + быстрый jump между фазами.
- Можно сделать кнопки `След. фаза` и `След. шаг`.

### Минусы
- Чуть сложнее реализации, чем Вариант 1.

---

## Какой вариант лучший
**Я бы выбрал Вариант 1 (линейный таймлайн событий)**.

Почему:
1. Он естественно покрывает ваш запрос «по-юнитово/по-фазово/по-ходово».  
2. Не тащит тяжелые полные копии состояния на каждый микрошаг.  
3. Проще масштабировать в будущем (анимации, фильтры, replay, экспорт логов).

## Предлагаемый MVP (минимально полезная версия)

### Шаг 1. Event-модель
Добавить структуру события:
- `turn`, `phase`, `side`, `unit_id`, `event_type`, `payload`, `ts`.

### Шаг 2. Сбор событий только для AI-side
- Для player-side поведение не меняем.
- Для model-side пишем события в буфер `model_replay_events`.

### Шаг 3. UI-контролы
- Кнопки: `След. шаг`, `Авто`, `Пауза`.
- Лейбл текущего шага: `Ход X / Фаза Y / Юнит Z / Действие A`.

### Шаг 4. Отрисовка
- На `След. шаг` применяем следующее событие.
- Обновляем карту и панель статуса.
- Для `command` без действий — переход сразу к следующему meaningful event.

### Шаг 5. Логи (обязательно)
Писать явные строки в viewer-логи:
- почему шаг пропущен (`нет валидных целей`, `фаза без действий`, `юнит уничтожен` и т.д.).

---

## UX-детали, чтобы было «понятно с первого раза»
- Цветовая подсветка:
  - активный юнит модели — оранжевый,
  - цель — красный,
  - завершившие действие — приглушённый.
- Скорость autoplay (0.25x / 1x / 2x).
- Горячая клавиша `N` = `След. шаг`.

---

## Риски и как снизить
- **Риск:** расхождение между «событием» и фактическим стейтом.  
  **Решение:** валидация после применения события + debug assert в dev-режиме.
- **Риск:** слишком много микрошагов и «долго кликать».  
  **Решение:** кнопка `След. фаза` и autoplay.
- **Риск:** пустые/непонятные шаги.  
  **Решение:** обязательная причина в логе и человекочитаемая подпись в UI.
